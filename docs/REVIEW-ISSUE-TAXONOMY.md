# Review Issue Taxonomy

## Purpose

This page defines a lightweight taxonomy for review feedback issues so comments stay searchable and bounded.

## Suggested Labels

| Label | Meaning |
| --- | --- |
| `review:packet` | the issue is about a bounded review packet |
| `review:profile` | the issue concerns a profile mapping |
| `review:binding` | the issue concerns a binding draft |
| `review:domain` | the issue concerns a domain protocol draft |
| `review:conformance` | the issue concerns fixture or validator evidence |
| `review:governance` | the issue concerns governance or review workflow |
| `review:cosigner` | the issue is a cosigner-style review request |
| `review:design-partner` | the issue is a design-partner style review request |
| `area:mcp` | MCP-related feedback |
| `area:a2a` | A2A-related feedback |
| `area:trust` | auth / trust feedback |
| `area:payments` | payment feedback |
| `area:provisioning` | provisioning feedback |
| `area:commerce` | commerce feedback |

## Issue Shape

A review issue should include:

- the packet or artifact under review
- the exact question asked
- the requested decision
- the follow-up owner
- a link back to the packet template

## Rule

If a review issue cannot be answered without opening the whole suite, the issue is too broad and should be split.
