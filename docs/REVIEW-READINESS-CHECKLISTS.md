# Review Readiness Checklists

## Purpose

These checklists help decide whether a packet is ready for outside review or still needs internal hardening.

## Cosigner Readiness Checklist

Use this when asking for a bounded review from an ecosystem maintainer or strong domain reviewer.

- [ ] The artifact family is narrow enough to review in one pass
- [ ] The requested decision is written as a single question
- [ ] The current implementation status is stated plainly
- [ ] The current conformance status is stated plainly
- [ ] The open questions are explicit
- [ ] The non-goals are explicit
- [ ] The packet does not ask for whole-suite approval
- [ ] The follow-up path is durable

## Design-Partner Readiness Checklist

Use this when asking a potential adopter whether the framing or shape is actually usable.

- [ ] The audience is named
- [ ] The adoption question is concrete
- [ ] The friction question is concrete
- [ ] The packet avoids overclaiming implementation maturity
- [ ] The packet avoids spec-level argument sprawl
- [ ] The packet has a clear follow-up owner

## Internal Review Readiness Checklist

Use this before turning a draft into a public packet.

- [ ] The relevant map or explainer page exists
- [ ] The doc links resolve
- [ ] The scope boundary is visible
- [ ] The review ask can be answered in one paragraph
- [ ] The packet matches the current repo state
